## Overall Rating: 7.5/10 (for an alpha)

The engineering is ahead of the design. The architecture is 8.5/10 but the UI/UX is closer to 7 — good bones, professional dark aesthetic, but inconsistencies and missing polish that separate it from shipping software.

---

## 5 THINGS BOOJY DOES WELL

### 1. The layout is immediately familiar
Three-panel DAW layout (Library | Timeline | Mixer) with bottom editor. Anyone who's used Ableton, Logic, or FL Studio can sit down and know where things are. The resizable dividers with collapse/expand are smooth. The 6-level responsive density system on the transport bar is genuinely clever — most DAWs just clip or overflow.

### 2. The piano roll velocity editing is best-in-class
The FL Studio-style velocity circles in the velocity lane are more intuitive than Logic's stacked bars or Ableton's thin lines. Click and drag is natural. The scale highlighting with out-of-scale note dimming is a feature Logic charges money for.

### 3. Recording workflow is surprisingly complete for alpha
Count-in with song context (you hear the actual mix during count-in, not just clicks), automatic overlap handling (trims/splits/deletes conflicting clips), and single Cmd+Z to undo an entire recording session. This is more polished than GarageBand's recording flow and approaches Logic's.

### 4. The color system is cohesive
The blue-tinted dark palette (#040412 editor, #2C2C32 chrome) feels intentional and premium. Track auto-colors (red=drums, orange=bass, green=synth, blue=guitar, purple=vocals) match what users expect. The accent blue (#40B3E8) is used sparingly and consistently for active/selected states.

### 5. Clip operations are solid
Split at playhead (Cmd+E), consolidate (Cmd+J), duplicate (Alt+drag), trim edges, snap to grid — the core clip workflow works. The automatic overlap resolution when recording or duplicating is a detail most DAWs handle poorly.

## 5 THINGS THAT NEED WORK (with mockups)

### 1. Unified Button System — "Outlined Blue" Style

**The problem**: Transport bar buttons and piano roll buttons look completely different despite serving the same role (toggle/action).

**Current state:**
- Transport bar: `accent @ 30%` tinted background when active, 1px divider border
- Piano roll: Solid `accent` fill when active, `dark` fill when inactive, no border
- Padding, font size, icon size all differ between them

**Proposed unified style — "Outlined Accent":**

The idea: every toggle/action button across the entire app uses the **same component** with the **same 4 states**. The outline is the defining feature — it gives the button a clear boundary on the dark chrome without needing a heavy fill.

### 2. Transport Bar — 3 Alternative Layouts (single row, 54px)

The issue isn't the number of controls — it's visual grouping. All three options keep everything in one row but improve scannability.

**Chosen: "Cluster Wells + Size Hierarchy" — hardware console feel**

Each cluster sits in a subtle recessed "well" (1px inset border, darkest background). Like sections on a hardware mixing console. The wells create grouping without needing divider lines. Transport buttons are slightly larger (32px vs 26px) for a subtle size hierarchy on the most important controls.

```
54px ┌────────────────────────────────────────────────────────────────────────────┐
     │ [Audi●]                                                                    │
     │         ┌─well────────────────┐ ┌──well─────┐ ┌─well─────────────────────┐│
     │ [↩][↪]  │[Loop|→][Snap|B][♪|1]│ │[⏵] [⏺][⏹]│ │ 1.1.1 │ 120 BPM │ 4/4  ││
     │         └─────────────────────┘ └───────────┘ └──────────────────────────┘│
     │           26px buttons            32px buttons   interactive readouts      │
     │                                                            [Ready✓] [≡][?]│
     └────────────────────────────────────────────────────────────────────────────┘
```

**Why this approach:**

- **Wells use `darkest` bg (#13151C) with 1px `divider` border.** Chrome stays at `dark` (#2C2C32). The ~15% brightness difference creates grouping without divider lines.
- **Transport buttons at 32px** (up from 28px) gives a subtle size hierarchy — eyes find play/record/stop first — without the extreme "dumbbell" effect of 36px.
- **Readouts stay interactive** — position, tempo, and signature keep their drag/click/scroll affordances because they still live in styled containers, not plain text.
- **Ages well** — when adding send buttons or tempo automation toggles later, they go into the appropriate well with no layout redesign.
- **Proven pattern** — Ableton uses recessed wells for its transport cluster, FL Studio uses them for toolbar sections. Standard audio software grouping technique.

### 3. Mixer Panel — Chosen: Progressive Density with Horizontal Faders

Keep the current horizontal row layout. Two axes of adaptation:
- **Height** controls layout: 2-row (≥50px) or 1-row (<50px). Down to 16px for 50+ track overview.
- **Width** controls density: controls drop off progressively to protect fader width. The fader is always the dominant element — it grows as things drop.

#### Height: Two layout modes

```
HEIGHT       LAYOUT
──────       ──────
≥ 50px       2-ROW — current layout (icon, name, MSR, pan / automation, dB, fader)
16-49px      1-ROW — single row, controls adapt to available width
```

#### Width: Progressive drop order (same for both 1-row and 2-row)

Controls drop off in this order as width shrinks. Each drop gives more space to the fader:

| Priority | Control drops | Why | Space freed |
|----------|--------------|-----|-------------|
| 1st | Automation button | Toggle, secondary action | ~80px |
| 2nd | dB text (0.0 dB) | Fader position is enough visual cue | ~50px |
| 3rd | Pan knob | Rarely adjusted during arrangement | ~24px |
| 4th | R (record arm) | Set once, not toggled during mixing | ~24px |
| 5th | M/S buttons | State shown by row opacity (muted = dimmed) | ~40px |
| Never | Track name + fader | Always need to know which track and its level | — |

**Key principle**: The fader is the most important mixer control. Everything else sacrifices space for it.

#### Collapsed tracks in the timeline

At heights <24px, clips render as thin 2px colored bars (track color) instead of full waveforms/notes:

```
16px │▎ ████████   ██████████████   ███│
     │▎                                │
      ↑ clip 1     clip 2          clip3
```

At 16px × 50 tracks = 800px — fits in a typical window for full arrangement overview.

#### How to collapse/expand

| Trigger | Action |
|---------|--------|
| **Drag resize handle** below 50px | Track smoothly enters 1-row mode |
| **Double-click resize handle** | Toggle between 16px (collapsed) and previous height |
| **Right-click track** → "Collapse Track" | Set to 16px, remember previous height |
| **Right-click mixer header** → "Collapse All" | All tracks to 16px |
| **Cmd+Shift+Down** | Collapse selected track(s) |
| **Cmd+Shift+Up** | Expand selected track(s) to previous height |
